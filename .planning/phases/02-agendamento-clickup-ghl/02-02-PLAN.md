<objective>
Entregar a fatia vertical do fluxo de valor principal ponta-a-ponta para o caso happy-path de MÍDIA ÚNICA (Reels e Feed estático): detectar tasks elegíveis, resolver legenda+mídia (com fallback para a mãe), baixar e descompactar o zip do MinIO com guardas de segurança, fazer upload da mídia ao GHL, criar o post agendado, e devolver `agendado` + GHL Post ID ao ClickUp. Output: pipeline batch executável (`npm start`) que agenda posts de mídia única e sincroniza o status de volta.
</objective>

## Phase Goal

**As a** gestor de conteúdo da Auton, **I want to** que uma task marcada `a agendar` com `Data de publicação` preenchida vire automaticamente um post agendado no GHL Social Planner do Instagram `auton.app` e volte como `agendado` com o id do post salvo, **so that** eu nunca precise copiar legenda, mídia e horário à mão entre o ClickUp e o GHL.

## Acceptance Criteria (fatia deste plano)
- [ ] `npm start` detecta tasks `a agendar` + data e ignora as demais (SCH-01)
- [ ] Tasks com GHL Post ID já preenchido são puladas (SCH-06)
- [ ] Legenda+mídia resolvidas com fallback para a mãe (SCH-02)
- [ ] Mídia (1 arquivo) baixada da URL pre-signed/pública do MinIO sem credenciais (D-08), descompactada com guardas, e enviada à media library do GHL no fluxo download→extract→upload (D-09) (SCH-03)
- [ ] Post de mídia única agendado no GHL no horário correto (SCH-04)
- [ ] Sucesso → task `agendado` + GHL Post ID gravado (SCH-05)

## Interface Context (contratos que este plano cria e o Plano 03 consome)
- `src/lib/zip.js` exporta: `downloadAndExtract(zipUrl) → Promise<{ files: Array<{name,path,buffer}>, tmpDir: string }>`, `cleanupTmp(tmpDir) → Promise<void>`, `mimeFromFilename(name) → string`. - `src/clients/ghl.js` ganha `uploadMedia(fileBuffer, fileName, mimeType) → Promise<{url, fileId}>`; `createPost(payload)` permanece a mesma assinatura mas recebe o payload real.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MinIO (URL do campo Link do post) → processo | Download de zip de fonte externa controlada por dado de task; entrada não confiável |
| zip → filesystem (tmpDir) | Conteúdo de arquivo arbitrário extraído para o disco local |
| processo → GHL API | Upload multipart + createPost cruzam para serviço externo |
| processo → logs | Risco de vazar URL pre-signed / tokens em log |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | SSRF (Server-Side Request Forgery) | `downloadAndExtract` fetch da URL do MinIO | mitigate | Validar `new URL(zipUrl).protocol === 'https:'` ANTES do fetch; lançar erro de validação; teste assegura que fetch não é chamado em protocolo inválido (HIGH) |
| T-02-05 | Denial of Service (zip-bomb / download gigante) | `arrayBuffer()` antes da descompressão | mitigate | Teto `MAX_DOWNLOAD_BYTES = 100MB` checado em `buf.byteLength` antes de instanciar AdmZip |
| T-02-06 | Tampering / Elevation of Privilege (zip-slip) | extração de entries do AdmZip | mitigate | `basename(entryName)` + guarda `relative(tmpDir, resolve(tmpDir,name))` não começa com `..` nem é absoluto; descartar entries maliciosas; teste com `../` (HIGH) |
| T-02-07 | Tampering (conteúdo não-zip) | buffer baixado | mitigate | Validar magic bytes `PK\x03\x04` antes de passar ao AdmZip |
| T-02-03 | Information Disclosure | logs do pipeline/zip | mitigate | NUNCA logar `zipUrl`/link do MinIO nem tokens; logar só taskId + fileName; AppError.fromGHL já filtra body |
| T-02-08 | Information Disclosure | upload multipart | mitigate | Reusa authHeaders sem logar Authorization (Fase 1, T-01-02); FormData sem Content-Type manual evita malformação |
</threat_model>
